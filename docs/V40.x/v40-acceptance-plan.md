# V40 Acceptance Plan

Date: 2026-06-30

## Acceptance Decision

Every V40 phase must produce one of:

- `passed scoped`: all phase-specific requirements passed with real evidence;
- `blocked`: external/local prerequisite unavailable with stable reason;
- `failed`: prerequisite available but behavior or quality violates the spec.

## User-Visible Acceptance

The user should be able to inspect a V40 report and answer:

- What source sample was used?
- What did V39 produce for the same sample?
- What did the no-WebUI V40 route produce?
- Was the candidate generated by Direct Local Runner or imported under an explicit manual
  fallback?
- Is the V40 output visibly more appealing and more action-readable than V39?
- Can the accepted candidate be previewed, applied to the target pet, and
  rolled back?
- Which candidates were blocked and why?

## Quality Gates

- Same-sample identity must remain recognizable.
- At least eight actions must be present and readable at desktop-pet scale.
- The result must not be a full-image transform-only animation.
- The result must not be a photo card, visible test label, border overlay, or
  raw generated image dump.
- Visual preference must compare V40 against V39 for the same sample.
- Human-facing evidence must include screenshots or rendered visual artifacts,
  not only JSON summaries.

## Tool Gates

- Direct Local Runner/model smoke must be real before generation can be claimed.
- Ollama smoke may support prompts/review only; it cannot approve assets.
- ComfyUI remains blocked and is not an active V40 pass dependency.
- WebUI remains blocked and is not an active V40 pass dependency.
- Missing model, missing API, VRAM failure, timeout, unsafe output, or
  unavailable local service must be recorded as blocked/failed.
- If Direct Local Runner is blocked, only explicit manual/import assets can continue, and the
  final claim must name that route.
- Current V40.3, V40.3R, and V40.3R2 evidence is not sufficient for V40.4
  entry: prompt-only candidates failed visual review, img2img candidates failed
  visual review, V40.3R identity-conditioned generation was initially blocked by
  runner-stack compatibility, and V40.3R2 identity-conditioned repair generated
  real candidates that failed explicit visual review.
- V40.3R3 may only decide the next candidate source. It cannot approve V40
  asset quality, cannot start V40.4 by itself, and cannot repeat failed V40.3R2
  outputs as accepted inputs.
- V40.3R4 may only replan candidate-source strategy after V40.3R3 blocked. It
  cannot approve host synthetic template GIFs, process-only UI checks, or failed
  prior candidates as accepted V40 assets.
- V40.3R5 may only audit the selected direct-runner route before generation. It
  cannot approve generated assets.
- V40.3R6 may only generate bounded candidate frame sequences after V40.3R5
  passes. It cannot unlock V40.4 unless at least two same-sample candidates pass
  explicit visual review.

## Security And Claim Gates

Evidence must not contain:

- token or Authorization values;
- raw runner payloads;
- raw prompts;
- raw local absolute paths;
- raw generated image bytes;
- raw photo bytes;
- EXIF/GPS;
- workspace path, config path, or credential path;
- unredacted terminal titles or command transcripts.

Forbidden ready claims:

- Petdex parity achieved;
- automatic photo-to-animation ready for arbitrary cats;
- automatic photo-to-2D ready for arbitrary cats;
- provider integration verified;
- Route B verified without real same-sample assets;
- 3D ready;
- production signed release ready;
- Windows ready;
- cross-platform ready;
- MCP ready;
- Claude Code integration verified;
- OS-level Codex window binding ready;
- all Codex workflows verified.
