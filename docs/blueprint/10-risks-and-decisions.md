# 风险与技术决策

## 主要风险

跨平台透明窗口：

- 风险：macOS 和 Windows WebView 透明行为不一致。
- 建议：MVP 使用小尺寸透明窗口，不做全屏 overlay；尽早双平台验证。

点击穿透：

- 风险：平台 API 差异大，会破坏拖拽和逗猫。
- 建议：MVP 不默认支持，只保留后续实验开关。

macOS 权限：

- 风险：辅助功能、输入监听、登录启动可能触发权限问题。
- 建议：MVP 不做全局鼠标 hook，不做登录启动。

Windows 置顶窗口行为：

- 风险：全屏应用、UAC、安全桌面不可控。
- 建议：允许关闭置顶，不承诺覆盖所有系统级窗口。

动画资源生产：

- 风险：动作资源制作周期可能超过工程开发。
- 建议：MVP 使用占位 sprite，先固化资产规范。

体验主线偏移：

- 风险：开发过程被 HTTP、MCP、CLI、USB 等工程模块牵引，最后做成技术正确但用户不想常驻桌面的通知器。
- 建议：每个阶段都用“猫咪是否更像可陪伴的桌宠”验收；普通状态默认动作为主，文本和声音克制使用。

照片自定义猫咪：

- 风险：用户期待照片直接变完整动态猫。
- 建议：MVP 不做照片自定义；模板套色、3D 和 Live2D 全部推迟。

MCP/CLI 接入复杂度：

- 风险：入口太多导致行为不一致。
- 建议：所有入口最终调用本地 HTTP API，`pet-protocol` 是唯一协议来源。

USB 串口设备兼容性：

- 风险：芯片、驱动、端口名、权限差异大。
- 建议：第一版不做 USB 真实硬件，也不做 mock adapter；有目标硬件后再实现真实 adapter。

打包签名：

- 风险：macOS Gatekeeper、Windows Defender、企业环境限制。
- 建议：MVP 先本地打包，公开发布前处理签名、公证和自动更新。

本地 API 安全：

- 风险：本机其他进程伪造事件。
- 建议：必须使用 token，只监听 `127.0.0.1`，严格校验 schema，动作/声音/灯效白名单。

事件风暴：

- 风险：agent 高频写入导致 UI 抖动和声音轰炸。
- 建议：队列上限、source 限流、重复合并、声音冷却。

## 第一版应该做

- Tauri 2 桌面透明猫窗口。
- 默认 2D sprite 猫。
- idle、walk、sleep、自主游走等基础生命感。
- 鼠标关注和基础逗猫。
- 拖拽、置顶、托盘、静音、退出。
- 本地 HTTP API。
- `PetEvent` schema 校验。
- 事件队列和状态机。
- `petctl notify`。
- 默认声音白名单。

## 第一版不应该做

- 完整 3D 猫。
- 照片自动生成动态模型。
- Live2D 编辑器。
- 全局鼠标 hook。
- 默认点击穿透。
- 真实 USB 硬件全量适配。
- USB mock 硬件。
- 完整 MCP server。
- Codex / Claude Code Skill 完整包。
- 照片自定义猫咪。
- 团队协作中枢。
- 自动更新和正式签名发布。
- 在线资产下载。
- 强状态仪表盘或通知中心。
- agent 直接控制 UI 或传资源路径。

## 推迟功能

- Rive / Live2D / GLTF renderer。
- 真实串口氛围灯。
- USB mock adapter。
- MCP server。
- Codex / Claude Code Skill 完整包。
- 照片自定义猫咪。
- 团队协作中枢。
- 自动更新。
- macOS 登录启动。
- 复杂多屏跨屏游走。
- 资产市场和云生成服务。
- 高级行为 AI，例如猫咪自主情绪系统。

## 最终技术负责人建议

第一版要做成一只开发者愿意常驻桌面的猫，而不是通用桌宠、通知中心、复杂角色生成器或 agent 状态仪表盘。核心价值是把 AI Agent 后台状态转译为低打扰桌面反馈。协议、安全、HTTP 和 petctl 都是支撑能力，不能压过桌宠本体体验。

第一条实现任务应从 Tauri 桌面壳开始：先做出可见、可拖、可常驻、带托盘的占位猫。MCP、skills、USB、照片自定义和高质量资产系统全部等 MVP 链路稳定后再推进。
